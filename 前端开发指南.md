# Vue 3 + TypeScript + JSX + Pinia 标准化前端开发规范指南 (2026版)

## 1. 技术栈与架构原则

### 1.1 核心技术栈版本
*   **框架核心**: Vue 3.x (最新稳定版)
*   **构建工具**: Vite 5.x / 6.x
*   **路由管理**: Vue Router 6.x
*   **状态管理**: Pinia (最新版)
*   **语言超集**: TypeScript 5.x+ (严格模式)
*   **模板语法**: JSX / TSX (推荐用于复杂逻辑组件) 或 `<script setup>`
*   **UI 组件库**: Ant Design Vue 4.x+ (按需加载)
*   **开发调试**: Vue DevTools 6.x

### 1.2 架构设计原则
1.  **Type-Safety First (类型安全优先)**
    *   全项目强制开启 TypeScript 严格模式 (`strict: true`)。
    *   **严禁**使用 `any` 和 `unknown` 类型，必须定义明确的接口 (Interface) 或类型别名 (Type)。
    *   所有函数必须显式声明返回值类型。
2.  **Composition API Only (组合式 API 唯一)**
    *   全面采用 Composition API (`<script setup>` 或 JSX 函数组件)。
    *   **禁止**使用 Options API (`data`, `methods`, `mounted` 等选项式写法)。
3.  **响应式规范**
    *   基础类型优先使用 `ref`，对象/数组优先使用 `reactive`。
    *   避免滥用 `toRef`/`toRefs`，仅在解构 Props 或 Store 状态时使用。
4.  **组件通信规范**
    *   建立清晰的数据流：单向数据流 (Props Down, Events Up)。
    *   跨层级/跨模块通信统一使用 Pinia Store。
    *   禁止使用 `$parent`, `$children`, `$refs` 进行直接组件实例操作。

## 2. 项目目录结构规范

采用基于功能域 (Feature-based) 与类型混合的目录结构，适用于中大型项目。

```text
src/
├── assets/                 # 静态资源 (images, fonts, styles)
├── components/             # 全局公共组件
│   ├── base/               # 基础原子组件 (Button, Input, Modal等)
│   ├── business/           # 业务通用组件 (UserCard, ProductList等)
│   └── layout/             # 布局组件 (Header, Sidebar, Footer)
├── composables/            # 组合式函数 (Hooks)
│   ├── useAuth.ts          # 认证相关逻辑
│   ├── useTable.ts         # 表格逻辑复用
│   └── index.ts            # 统一导出
├── config/                 # 项目配置文件
│   ├── env.ts              # 环境变量封装
│   └── settings.ts         # 全局配置项
├── directives/             # 自定义指令
├── hooks/                  # (可选) 兼容旧习惯的 hooks 目录，建议合并至 composables
├── layouts/                # 页面布局模板
├── router/                 # 路由配置
│   ├── index.ts            # 路由入口
│   ├── routes.ts           # 路由表定义
│   └── guards.ts           # 路由守卫
├── services/               # API 服务层
│   ├── api/                # 具体模块 API 定义
│   ├── http.ts             # Axios/Fetch 实例封装
│   └── types.ts            # API 请求/响应类型定义
├── stores/                 # Pinia 状态管理
│   ├── modules/            # 模块化 Store
│   │   ├── user.ts
│   │   └── app.ts
│   └── index.ts
├── styles/                 # 全局样式
│   ├── variables.scss      # SCSS 变量
│   ├── mixins.scss         # SCSS 混入
│   └── global.scss         # 全局重置样式
├── types/                  # 全局 TypeScript 类型定义
│   ├── global.d.ts
│   └── common.ts
├── utils/                  # 工具函数库
│   ├── format.ts           # 格式化函数
│   ├── validate.ts         # 验证函数
│   └── storage.ts          # 本地存储封装
├── views/                  # 页面级组件 (按业务模块划分)
│   ├── dashboard/
│   ├── user-management/
│   └── login/
├── App.vue                 # 根组件
└── main.ts                 # 应用入口
```

### 命名规范
*   **文件夹**: 统一使用复数形式 (e.g., `components`, `services`)，_kebab-case_ (短横线命名)。
*   **组件文件**: _PascalCase_ (大驼峰命名)，如 `UserProfile.tsx` 或 `UserProfile.vue`。
*   **组合式函数**: 必须以 `use` 开头，_camelCase_ (小驼峰命名)，如 `useAuth.ts`。
*   **业务组件**: 若属于某父组件的子集，以父组件名为前缀，如 `TodoList`, `TodoListItem`。
*   **Store**: 文件名与 State 名称一致，如 `user.ts` 对应 `useUserStore`。

## 3. TypeScript 与组件开发规范

### 3.1 脚本与类型定义
*   **JSX/TSX 模式**:
    *   推荐使用 `.tsx` 后缀文件编写组件，利用 TypeScript 强大的类型推断。
    *   若使用 `.vue` 文件，必须使用 `<script setup lang="ts">`。
*   **Props 与 Emits**:
    *   **TSX**: 定义 Interface 继承 `HTMLAttributes` 或自定义 Props 接口，通过泛型传递给组件。
    *   **Vue SFC**: 必须使用 `defineProps<T>()` 和 `defineEmits<T>()` 泛型语法，**禁止**使用运行时对象定义 Props。
    *   复杂 Prop 类型必须使用 `PropType<YourType>` 进行断言。
*   **组合式函数 (Composables)**:
    *   必须显式定义返回值的 Interface 或 Type。
    *   禁止返回隐式 `any` 类型的对象。
    *   参数必须包含类型注解。
*   **Pinia Store**:
    *   使用 `defineStore('id', { ... })` 时，State, Getters, Actions 必须明确类型。
    *   推荐直接使用 Setup Store 风格 (`const useUserStore = defineStore('user', () => { ... })`) 以获得更好的类型推断。

### 3.2 组件开发最佳实践
*   **逻辑简化**:
    *   优先使用 `v-if` / `v-else` 或 JS 三元表达式处理条件渲染，避免嵌套过深的 `v-if`。
    *   在 TSX 中利用 JavaScript 原生的逻辑控制流。
*   **列表渲染**:
    *   必须为 `v-for` 或 `.map()` 渲染的列表项提供唯一的 `:key`。
    *   大数据量列表 (超过 50 项) 必须使用虚拟滚动 (Virtual Scroller)。
*   **生命周期**:
    *   优先使用组合式 API 钩子 (`onMounted`, `onUnmounted`, `onBeforeUpdate` 等)。
    *   DOM 操作必须在 `onMounted` 中执行。
    *   定时器、事件监听器等副作用必须在 `onUnmounted` 中清理，防止内存泄漏。
    *   **禁止**在 `setup` 函数外部或 `created` (Options API) 中执行异步初始化逻辑。
*   **样式隔离**:
    *   组件样式默认开启 `scoped`。
    *   全局样式仅定义在 `styles/global.scss` 中。
    *   推荐使用 CSS Variables 进行主题化配置。

## 4. 工程化配置规范

### 4.1 代码质量工具
*   **ESLint**:
    *   启用 `plugin:vue/vue3-recommended` 和 `@typescript-eslint/recommended`。
    *   强制规则：
        *   `@typescript-eslint/explicit-function-return-type`: 强制函数返回类型。
        *   `vue/no-options-api`: 禁止 Options API。
        *   `vue/require-explicit-emits`: 强制定义 Emits 类型。
        *   `@typescript-eslint/no-explicit-any`: 禁止 `any`。
*   **Prettier**:
    *   统一代码格式（缩进、引号、分号等），与 ESLint 配置协同工作。
*   **vue-tsc**:
    *   在 CI/CD 流程和 `build` 命令前强制执行类型检查，确保无类型错误方可构建。

### 4.2 构建配置 (Vite)
*   **代码分割 (Code Splitting)**:
    *   配置 `manualChunks`，将 `node_modules` 中的大型依赖 (如 vue, pinia, ant-design-vue) 单独拆包。
    *   按业务模块动态导入 (Dynamic Import) 实现细粒度拆分。
*   **路由懒加载**:
    *   所有路由组件必须使用 `() => import('@/views/...')` 方式引入。
*   **CSS 优化**:
    *   开启 CSS 代码拆分，提取公共样式。
    *   配置 PostCSS 插件进行自动前缀补全和压缩。
*   **按需加载**:
    *   配置 UI 组件库 (如 Ant Design Vue) 的按需引入插件，减少打包体积。

## 5. 性能优化与工程化实践

### 5.1 构建性能
*   **Tree Shaking**: 确保所有导入均为 ES Module 格式，充分利用 Vite 的 Tree Shaking 能力。
*   **预加载/预获取**: 对关键路由资源使用 `<link rel="prefetch">` 或 `<link rel="preload">`。
*   **图片优化**: 使用现代格式 (WebP/AVIF)，配置 Vite 插件进行图片压缩和 CDN 替换。

### 5.2 运行时性能
*   **响应式优化**:
    *   避免在大对象上使用 `reactive`，尽量拆分小对象或使用 `ref`。
    *   对于不需要响应式的大数据集，使用 `shallowRef` 或 `markRaw`。
*   **渲染优化**:
    *   频繁切换显示状态的元素使用 `v-show` 替代 `v-if`。
    *   计算属性 (`computed`) 缓存耗时计算逻辑，避免在模板中直接调用复杂函数。
*   **事件处理**:
    *   使用事件委托处理大量子元素的事件。
    *   高频触发事件 (scroll, resize, input) 必须使用防抖 (Debounce) 或节流 (Throttle)。

## 6. 总结

本规范旨在构建**高可维护性**、**强类型安全**且**高性能**的 Vue 3 前端应用。通过强制执行 TypeScript 严格模式、全面采用 Composition API (JSX/SFC)、细化目录结构以及实施严格的工程化检查，团队能够有效降低技术债务，提升开发效率，确保代码质量符合 2026 年及未来的行业标准。

所有新项目的初始化、现有项目的重构以及代码审查 (Code Review) 均应严格遵循本指南。
